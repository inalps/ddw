The big idea: docs in your repo, gates on the way to main

Most AI tools just write code. DDW makes you write down WHAT you're building, WHY, and HOW you'll know it's done — all as plain markdown files inside your repo. Then it puts hard gates between phases. Nobody ships work that skipped a step. Not you. Not your AI helper.

🚧 No PRD? → No decision.
🚧 No decision? → No task.
🚧 AC failing? → No close.
🚧 Not closed? → No merge.

Six months from now, anyone reading your repo will know exactly what was decided, when, and why. That's the gold DDW protects.

Three things DDW tracks 📚

Every project ends up with three kinds of docs in workflows/. Here's what each one is for.

💭

PRD — Product Requirements Doc

The WHAT and the WHY.

workflows/prds/PRD-{date}-{name}.md

What goes inside: what you want to build, why it matters, what success looks like, scope limits. Like a proposal you'd write before a big school project.

Status flow: draft → ready → closed

Built by /ddw:ideate — the AI interviews you with smart questions, then writes the PRD.

🧭

Decision — Architecture Decision Record (ADR)

The HOW.

workflows/decisions/DEC-{date}-{name}.md

What goes inside: context, options considered, decision made, trade-offs, expected consequences. Like a lab notebook explaining "we tried approach A, B, and C — we picked B because…".

Status flow: proposed → decided → archived

Built by /ddw:decision — the architect-AI reviews trade-offs and writes the record with you.

✅

Task — Work unit with QA log

The DO.

workflows/tasks/TASK-{date}-{name}.md

What goes inside: scope, files you'll touch, Acceptance Criteria table (each AC has a check type: code-grep, code-review, spec-compare, manual), Work Log (timestamped progress), Review Log (where /ddw:qa writes pass/fail/regression findings). Like a homework assignment with a checklist AND a grading rubric.

Status flow: planned → in_progress → review_and_bugfix → done → archived

(In team-PR mode: an extra in_review state slots in between done and archived while the PR is open.)

Built by /ddw:task. Executed by /ddw:sendit. Reviewed by /ddw:review (which runs /ddw:qa). Closed by /ddw:close.

Setup in 30 seconds ⚡

Three small steps. Done before your coffee cools.

1

Clone the DDW repo

Grab the latest DDW from GitHub and put it somewhere on your machine. Anywhere is fine — your home folder, a tools folder, wherever you keep cloned repos.

git clone …
2

Start Claude Code with the plugin

From your project folder, launch Claude Code and point it at the DDW folder you just cloned.

$ claude --plugin-dir /path/to/ddw
3

Bootstrap DDW into your project

Inside Claude Code, run the init command. DDW will create a workflows/ folder for your decisions, tasks, and rules.

/ddw:init
What just happened: Your project now has a workflows/ folder. Everything DDW tracks — decisions, tasks, guardrails — lives there as plain markdown. You can read it with any editor. Git tracks it like normal code.

The flow 🚀

Five steps from idea to shipped. Each step is one slash command.

1

Have an idea? Shape it.

Got a rough idea? /ddw:ideate asks you smart questions until it becomes a clear plan (a PRD).

/ddw:ideate
2

Make the call.

A built-in "architect" reviews your plan and writes down the technical decision. No more "wait, why did we choose that?"

/ddw:decision
3

Break it into tasks.

The decision becomes small, focused tasks. Each task has a goal, files to touch, and how to know it's done.

/ddw:task
4

Ship it!

/ddw:sendit creates an isolated workspace for the task and the AI starts building. You watch. You guide. Tests get run automatically.

/ddw:sendit
5

Close the loop.

QA runs. You check the work. /ddw:close merges to main, updates the spec, archives the task. Done!

/ddw:close
Stuck somewhere? Just type next? in Claude Code. DDW will look at where you are and tell you what to do.

The magic: Auto mode 🌙

/ddw:auto

Got a stack of tasks lined up? Type /ddw:auto before bed.

DDW will work through them one by one — implementing, testing, reviewing — while you sleep. When you wake up, the work is done (or queued for your input if something needed a human call).

It even runs tasks in parallel when they don't conflict. If a task needs the database in a sensitive way, DDW knows to wait its turn. Smart enough not to crash into itself. 💪

What makes DDW different ✨

Plain markdown, all the way

No proprietary DB, no SaaS dashboard. Your decisions and tasks are .md files in your repo — readable, diffable, greppable, forever.

🛡️

Guardrails built-in
