大事なアイデア：設計はリポジトリの中に、mainへの道にゲートあり

他のAIツールはコードを書いて祈るだけ。DDWは違う。何を作るか、なぜ作るか、どうやって完了を判断するか — 全部リポジトリの中にマークダウンで書かせる。そして各フェーズの間にハードゲート。ステップを飛ばしたコードは、誰のものでもmainに行けない。あなたのも、AIのも。

🚧 PRDなし? → 決定できない。
🚧 決定なし? → タスク作れない。
🚧 AC失敗? → クローズできない。
🚧 未クローズ? → マージできない。

6ヶ月後、リポジトリの誰が見ても、何が決まって、いつ、なぜそうしたかが分かる。それがDDWが守る「金」。

DDWが追跡する3つのもの 📚

どのプロジェクトも workflows/ に3種類のドキュメントが溜まっていく。それぞれ何のためのものか、見てみよう。

💭

PRD — 要件定義書

「何を」「なぜ」を記録。

workflows/prds/PRD-{date}-{name}.md

中身: 何を作りたいか、なぜ大事か、成功とは何か、スコープの範囲。大きい学校プロジェクトの前に書く企画書みたいなもの。

ステータスの流れ: draft → ready → closed

作成: /ddw:ideate — AIが質問してくれて、答えを元にPRDを書く。

🧭

Decision — アーキテクチャ決定記録 (ADR)

「どうやって」を記録。

workflows/decisions/DEC-{date}-{name}.md

中身: コンテキスト、検討した選択肢、決定、トレードオフ、予想される結果。「A, B, Cを試した — Bを選んだ理由は…」と書く実験ノート的なもの。

ステータスの流れ: proposed → decided → archived

作成: /ddw:decision — アーキテクトAIがトレードオフをレビューして一緒に書く。

✅

Task — QAログ付きの作業単位

「実行」を記録。

workflows/tasks/TASK-{date}-{name}.md

中身: スコープ、触るファイル、受入基準(AC)テーブル(各ACにチェックタイプ: code-grep / code-review / spec-compare / manual)、作業ログ(タイムスタンプ付き進捗)、レビューログ(/ddw:qa がPASS/FAIL/REGRESSIONを追記)。チェックリスト＋採点ルーブリック付きの宿題みたいなもの。

ステータスの流れ: planned → in_progress → review_and_bugfix → done → archived

(チームPRモードでは in_review が done と archived の間に入る。)

作成: /ddw:task、実行: /ddw:sendit、レビュー: /ddw:review (内部で /ddw:qa が走る)、クローズ: /ddw:close。

30秒でセットアップ ⚡

3ステップだけ。コーヒーが冷める前に終わる。

1

DDWリポジトリをcloneする

GitHubからDDWの最新版を取得して、好きな場所に置く。ホームフォルダでも、ツール用フォルダでも、どこでもOK。

git clone …
2

Claude Codeをプラグイン付きで起動

プロジェクトフォルダから、Claude Codeを起動。さっきcloneしたDDWフォルダのパスを指定する。

$ claude --plugin-dir /path/to/ddw
3

プロジェクトに導入

Claude Code内で初期化コマンドを実行。workflows/フォルダが作成されて、決定・タスク・ルールがここに溜まっていく。

/ddw:init
これで何ができた? プロジェクトに workflows/ フォルダができた。DDWが管理する決定・タスク・ガードレールは、全部ここにマークダウンで保存される。普通のエディタで読めるし、Gitもいつも通り追跡する。

開発の流れ 🚀

アイデアから出荷まで5ステップ。各ステップはスラッシュコマンド1つ。

1

アイデアを整える

ふんわりしたアイデアでもOK。/ddw:ideate が質問してくれて、ちゃんとした企画書(PRD)に仕上がる。

/ddw:ideate
2

方針を決める

内蔵の「アーキテクト」が企画をレビューして、技術的な決定を記録する。「なんでこの方針にしたっけ?」がなくなる。

/ddw:decision
3

タスクに分ける

決定が、小さくて明確なタスクに分解される。各タスクには目標、触るファイル、完了の条件が書かれてる。

/ddw:task
4

送り出す!

/ddw:sendit がタスク用の隔離されたworktreeを作って、AIが実装を始める。あなたは見守る、必要なら指示する。テストも自動。

/ddw:sendit
5

クローズして完了

QAが走る。あなたが最終チェック。/ddw:close がmainにマージして、仕様を更新して、タスクをアーカイブ。完了!

/ddw:close
迷ったとき? Claude Codeで next? って入力するだけ。DDWが今どこにいるかを見て、次にやることを教えてくれる。

魔法のような自動モード 🌙

/ddw:auto

タスクが溜まってる? 寝る前に /ddw:auto って打つだけ。

DDWが1つずつタスクを処理してくれる — 実装、テスト、レビュー、あなたが寝てる間に。朝起きたら作業が終わってる(人間の判断が必要な部分はキューに残る)。

しかも、競合しないタスクは並列でこなす。 データベースに敏感な変更が走るときは、ちゃんと順番待ちする。 自分同士でぶつからない賢さ。💪

DDWの「ここがちがう」✨

📜

全部マークダウン

独自DBなし、SaaSダッシュボードなし。決定もタスクも、リポジトリの中の .md ファイル。読める、diffできる、grepできる、ずっと残る。

🛡️

ガードレール内蔵

プロジェクトのルールを一度書けば、DDWが毎回の変更でチェック。AIがあなたの基準をスルーできない。

📦

1タスク1ブランチ

巨大なブランチで混乱しない。各タスクが小さくて集中してて、レビューしやすい。トランクベース開発、ちゃんとやる版。

🔐

Opusでセキュリティ監査

/ddw:audit でOWASP+STRIDEのセキュリティレビューが走る。ユニットテストが見つけられないバグも発見。

🌏

日本語OK

「日本語で進めて」って言うだけで切り替わる。ドキュメントもプロンプトも。設定不要。

🆓

無料・オープンソース

MITライセンス。フォークOK。チームのワークフローに合わせて改造OK。ベンダーロックなし。

今すぐ試してみよう 🎯

  1. 好きなプロジェクトでClaude Codeを開く(または新規作成 — mkdir hello-ddw && cd hello-ddw && git init)。
  2. /ddw:init でworkflowsフォルダをセットアップ。
  3. /ddw:ideate を実行して、作りたいものを伝える。ボタン1つでも、スクリプトでも、何でもOK。
  4. そのまま流れに従う。各コマンドが次にやることを教えてくれる。
